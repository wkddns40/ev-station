# Migration story

EV-STATION was a 2023 portfolio project built on Create React App, a single 606-line `evstation.js` god component, and Mapbox tiles behind a hardcoded API key. This document records what changed, why, and what would be different next time.

The pre-refactor source lives on the `legacy` branch and the signed `v0-legacy` tag.

```
git fetch origin
git checkout legacy
```

## Why the refactor

The legacy app shipped. It worked. The reasons it had to be rewritten anyway:

1. **The Mapbox token was committed and public on GitHub.** It needed to be killed in a Phase 0 pass before *anything* else; the cleanest path was to migrate the whole stack while we were already touching git history.
2. **30+ `useState` hooks + sync `useEffect`s.** Adding a feature meant tracing which of the three shadow-state hooks was authoritative for a given field. Filter changes silently dropped results when an `if/else` cascade hit an unreachable branch (B3 in the regression suite).
3. **No tests, no CI, no types.** Any change was a roll of the dice. The first thing a future contributor would need is confidence the unit didn't already break — and the only artifact was a 606-line file with no narrative.
4. **Mapbox's free tier was tightening.** Migrating to OpenFreeMap (Liberty style, no key) removed both the leaked-secret risk and the per-month API budget.

## Decisions and rationale

Locked decisions live in [`REFACTOR_PLAN.md` §12](../REFACTOR_PLAN.md). The interesting ones get full ADRs:

- [ADR 001 — Vite over CRA](decisions/001-vite-over-cra.md). CRA was deprecated in 2023; bundle and dev-server speed weren't even the main argument.
- [ADR 002 — MapLibre over Mapbox](decisions/002-maplibre-over-mapbox.md). Drop-in style URL swap, no API key, identical UX for the user.
- [ADR 003 — useReducer + Context over Redux](decisions/003-usereducer-over-redux.md). State graph is one shape (filters); Redux would be ceremony for zero gain.
- [ADR 004 — TanStack Query for data fetching](decisions/004-tanstack-query.md). The legacy hand-rolled fetch was reinventing 80% of what Query gives you.

Other locked decisions worth noting:

- **D5 — backend stays dev-only.** The production demo is a static 400-feature GeoJSON snapshot generated by `backend/scripts/generate_mock.py` (seed 42, deterministic). Means there's no live DB to defend; the deployed surface is plain static files.
- **D9 — commit identity is `wkddns40 <wkddns40@gmail.com>` only.** No co-author trailers. The Phase 2 cleanup PR accidentally landed a Claude Code default `Co-Authored-By` trailer, surfaced a "Contributors 2" widget on the public repo, and triggered a force-push rollback + a GitHub Support ticket to GC the orphan commits + a project-wide rule in CLAUDE.md and AGENTS.md to block the trailer for every future commit. The full incident lives in the git log around `5270ae2` (orphan).
- **D11 — every commit is signed.** SSH ed25519 signing key, per-repo `git config gpg.format=ssh` + `commit.gpgsign=true`. Every commit since Phase 0 shows GitHub's "Verified" badge.

## Lessons learned

**What worked**

- **Phase-by-phase atomic-task playbook.** REFACTOR_PLAN.md broke the refactor into ~200 atomic ≤10m tasks with explicit verification checkboxes per phase. The plan was the source of truth; deviations got noted in the commit body. Phase 0 → Phase 5 covered ~25 work-hours of execution with no replanning mid-phase.
- **Locked decisions amendment gate.** After accidentally letting a Claude trailer through in Phase 2, we added a project-wide rule (CLAUDE.md + memory) that any §12 D-row edit must be surfaced to the user *before* writing it. Worked on the first test case (D8 amendment for the ev-charging → ev-station migration was held at the gate and the user declined to apply it).
- **`npm audit` as a refactor scorecard.** Going from 70 → 8 findings just by removing CRA was a clean before/after number.
- **Squash-merge per phase (or per sub-phase).** Clean linear history on master, one commit per logical unit, atomic revert points if anything regressed.

**What didn't**

- **Code-splitting `Evstation` itself regressed LCP by ~1.5s on mobile.** The map *is* the largest contentful element; deferring it past first paint just delays LCP. Reverted (`docs/lighthouse/README.md` records the gap). The deck.gl + maplibre WebGL stack on Lighthouse-throttled mobile genuinely caps Performance at ~56; reaching 80 needs an architectural lift (server-side render, lighter renderer) that's out of scope.
- **Husky `lint-staged` config landed broken** (the `bash -c 'cd frontend && ...'` wrapper doesn't resolve staged file paths after cwd changes). CI catches lint independently, so it's hygiene-only; flagged for Phase 5 follow-up.
- **CRA's `manifest.json` icon entries** (`logo192.png`, `logo512.png`) were never shipped, generating two 404s on every Vercel deploy until Phase 3 caught them.

**What I'd do differently**

- **Run `npm audit` and `git secret-scan` on commit 1, not commit 30.** The Mapbox token leak forced a Phase 0 hot-fix that could've been avoided by treating secrets as a pre-commit gate from day one.
- **Plan for the test stack before the feature work, not after.** Phase 4 wrote 62 tests for code that had already been refactored. Writing them in Phase 1 would've made the Phase 2 component split provably safe instead of provably *plausible*.
- **D2 — pick the map library after the Lighthouse target is set.** We picked MapLibre because it killed the API-key problem cleanly. We then discovered the WebGL stack made mobile Perf ≥ 80 effectively unattainable. The right order is: name the perf target first, then choose a renderer that can meet it.
